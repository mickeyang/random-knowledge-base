# Vibe Coding 实战方法论——基于智谱 GLM + Cline + VSCode 的单页 Web App 开发复盘

## 一、什么是 Vibe Coding

Vibe Coding（氛围编程）由 Andrej Karpathy 于 2025 年提出，核心理念是：**开发者用自然语言描述需求与验收标准，由大语言模型承担代码实现，人负责意图定义、方向引导和结果审查**，而非逐行手写代码[citation:8][citation:11]。本次使用智谱 GLM 大模型配合 VSCode + Cline 插件，完成了一个自包含（Self-contained）单页 Web Application 的原型开发，验证了这一范式在小型前端项目上的高效性。

## 二、准备阶段：需求与视觉先行（Spec-First）

本次实践中最关键的成功因素，是**提前用 Markdown 写好详细需求文档，并附上 JPEG 格式的 UI Mockup 图片一并提交给 AI**。这相当于传统流程中 Business Analyst 的角色——把"你想要什么"说清楚，AI 才能一次生成接近预期的布局与内容[citation:3][citation:14]。

最佳实践建议：
- **需求文档（requirements.md）**：明确技术栈（如原生 HTML/CSS/JS 或指定框架）、页面区块、交互逻辑、数据源及边界条件。
- **视觉参照（Mockup 截图）**：用 Figma 或手绘截图标注关键 UI 元素，让 AI 理解间距、配色、组件排布，减少样式返工。
- **约束声明**：写明"单文件 self-contained""不引入外部 CDN 依赖"等限制，防止 AI 擅自加打包工具或框架。

## 三、工具链与工作流

推荐组合：**Windows + Visual Studio Code + Cline（AI Agent 插件）+ 智谱 GLM（或同级代码能力强模型）**[citation:6][citation:9]。

Cline 的优势在于支持多文件读写、终端命令执行和任务记忆，适合 Agent 模式的"自主完成任务→反馈→再修正"循环。典型工作流为：

> 投喂需求文档 + Mockup → AI 生成初版代码 → 本地预览 → 把偏差/报错贴回对话 → AI 迭代修复 → 验收通过

## 四、上下文窗口管理与会话续接（关键踩坑点）

本次实践遇到典型问题：**多轮迭代后对话上下文（Context Window）超限，API 停止响应**。这是 Vibe Coding 的常见痛点——长对话累积 token 会导致模型"降智"甚至拒绝输出[citation:4][citation:6]。

应对方案：
1. **阶段性开新 Task**：功能模块告一段落或对话明显变长时，主动新开会话。
2. **Task ID 续接法**：记录上一 Task ID / 摘要，新会话要求 AI"读取上一任务的断点说明并接续"，将关键决策摘要（而非全量历史）注入新上下文，实现无缝继续[citation:7]。
3. **上下文分层**：项目级约定（目录结构、编码规范）写入项目根目录的规则文件（如 `.clinerules` 或 `AGENTS.md`），每次新会话自动加载，避免重复教学[citation:1][citation:14]。

原则：**小步快跑、早开新对话、用文档而非聊天记录传递记忆**。

## 五、验收与交付物

AI 不仅能写业务代码，还应要求其生成配套的工程文档。本次让 GLM 按指定格式输出 `README.md`，说明应用用途、设计理念、目录结构和使用方式——这对后期回顾及分享至关重要[citation:9][citation:12]。

验收时注意：
- 实际运行预览，核对 UI 与 Mockup 吻合度
- 检查是否有多余依赖或硬编码
- 要求 AI 补充 README 及必要注释，固化设计意图

## 六、Vibe Coding 方法论小结

综合本次经验与业界实践，可提炼出五条 Vibe Coding 核心原则[citation:3][citation:10][citation:14]：

| 原则 | 要点 |
|---|---|
| Spec-First | 先写需求+Mockup，再让 AI 动手 |
| 小步迭代 | 拆小任务，一次一个功能点 |
| 上下文工程 | 控制会话长度，新开任务用文档续接记忆 |
| 人机分工 | 人管意图/验收/架构约束，AI 管语法/样板 |
| 文档沉淀 | 让 AI 同步生成 README、注释、规则文件 |

Vibe Coding 不是"瞎蒙出代码"，而是**用工程化的方式管理 AI 的上下文与任务边界**，把自然语言变成真正的开发生产力。对于原型验证、Demo 和内工具场景，这一方法论能大幅压缩从想法到可运行产品的距离。
